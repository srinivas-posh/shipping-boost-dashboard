How This Works

Each listing in a search session is assigned a shipping bucket based on its average delivery speed (D1 = ships in ≤1 day, D2 = ≤2 days, …, D7+ = 8+ days, no_sale = no shipping history). The experiment boosts each listing's ranking score: new_score = base_score + boost[bucket] × importance. Faster-shipping listings get higher boosts and rise to earlier pages.

Within each search session, all listings are re-sorted by their new scores. Page bucket is set by rank: P1 = ranks 1–40, P2 = 41–80, …, P7 = 241–280. Because total impressions per page are fixed, every gain at P1 displaces an equal number of listings — delta column totals always equal zero. Row totals are also invariant (each listing belongs to exactly one bucket regardless of page).

Lifetime uses each listing's all-time average shipping speed; 90-Day uses the most recent 90 days (a more dynamic, recent signal).

📌 V1 defaults (from experiment): D1=1.00, D2=1.00, D3=0.90, D4=0.50, D5=0.30, D6–D7+=0.30, did_not_ship=0.30, no_sale=0.30 · Importance=1.00. At these settings, tables show actual experiment data from the sheet (50,000 impressions). Other configurations show live simulation (~49,880 impressions, ±1–5% approximation).

90-Day Shipping Time

avg_shipping_time_90d

Bucket assignment uses shipping speed over the most recent 90 days. More listings fall into no_sale (sellers inactive recently), and more into fast buckets (recently improved sellers). The 90-day signal tends to produce larger shifts for D1–D3, while the no_sale bucket loses significantly more P1 impressions. Compare the two views to understand recency effects.
